# Story E10-S2 — Configuration & Troubleshooting Docs

**Epic:** E10 — Documentation & Learner Experience
**Status:** Pending

---

## User Story

**As an** intermediate user (Audience B),
**I want** a full configuration reference and troubleshooting cards for each HPA failure code,
**so that** I can tune profiles and recover independently.

---

## Acceptance Criteria

- `docs/configuration.md`: all env vars, profiles, modes, defaults, memory requirements
- `docs/troubleshooting.md`: one card per HPA-301..306 with Symptom / Check / Fix / Retry fields (mirrors `print_failure_hint`)
- Both docs kept in sync with script behavior (single source of truth is script; docs mirror it)

---

## Tasks

### Task 10.2.1 — Write `docs/configuration.md`: all env vars with defaults, allowed values, profile table
- [ ] Create `docs/configuration.md`
- [ ] **Environment Variables** section: create a table with columns: Variable, Default, Allowed Values, Description — cover:
  - [ ] `APP_MODE`: default `landing`, values `landing|api`
  - [ ] `LOAD_MODE`: default `pod`, values `pod|host`
  - [ ] `PROFILE`: default `tiny`, values `tiny|balanced|stretch`
  - [ ] `NAMESPACE`: default `autoscaling-lab`, values any valid K8s namespace name
  - [ ] `APP_DEPLOYMENT`: default `autoscaling-lab-app`, values any valid deployment name
  - [ ] `HPA_RAMP_SEC`: default `120`, values positive integer (seconds)
  - [ ] `HPA_COOLDOWN_SEC`: default `180`, values positive integer (seconds)
  - [ ] `HPA_POLL_SEC`: default `10`, values positive integer (seconds)
  - [ ] `ARTIFACT_ROOT`: default `artifacts/<run-id>`, values any writable path
- [ ] **Profile Ladder** section: table with columns: Profile, maxReplicas, Min RAM, Use Case
  - [ ] tiny: 5, 2GB, laptops / CI / demo
  - [ ] balanced: 7, 4GB, development workstations
  - [ ] stretch: 10, 8GB, high-spec machines or cloud VMs
- [ ] **App Modes** section: brief description of `landing` (HTML, nginx) vs `api` (JSON endpoint) with expected load shape differences
- [ ] **Load Modes** section: brief description of `pod` (in-cluster busybox) vs `host` (local curl loop) with pros/cons

### Task 10.2.2 — Write `docs/troubleshooting.md`: 6 failure code cards matching `print_failure_hint` output format
- [ ] Create `docs/troubleshooting.md`
- [ ] Opening paragraph: explain that failure codes are generated by the HPA proof gate and that `./scripts/fix.sh HPA-30X` applies the canonical fix
- [ ] For each code 301–306, create an H2 section: `## HPA-30X — <title>` matching `print_failure_hint` output
  - [ ] **HPA-301**: Symptom: HPA resource not found after deployment. Check: `kubectl get hpa -n autoscaling-lab`. Fix: `./scripts/fix.sh HPA-301`. Retry: `./scripts/up.sh --resume hpa_proof`
  - [ ] **HPA-302**: Symptom: `kubectl top nodes` fails or returns no metrics. Check: `kubectl -n kube-system get pods -l k8s-app=metrics-server`. Fix: `./scripts/fix.sh HPA-302`. Retry: `./scripts/up.sh --resume hpa_proof`
  - [ ] **HPA-303**: Symptom: HPA shows `<unknown>/50%` for CPU. Check: `kubectl -n autoscaling-lab get deploy autoscaling-lab-app -o jsonpath='{.spec.template.spec.containers[0].resources.requests.cpu}'`. Fix: `./scripts/fix.sh HPA-303`. Retry: `./scripts/up.sh --resume hpa_proof`
  - [ ] **HPA-304**: Symptom: Load generator failed to start or exited immediately. Check: `./scripts/load.sh --status`. Fix: `./scripts/fix.sh HPA-304`. Retry: `./scripts/up.sh --resume hpa_proof`
  - [ ] **HPA-305**: Symptom: Replica count never exceeded baseline during ramp window. Check: `cat artifacts/<run-id>/hpa/replica_samples.csv`. Fix: `./scripts/fix.sh HPA-305`. Retry: `./scripts/up.sh --resume hpa_proof`
  - [ ] **HPA-306**: Symptom: Replicas did not decrease within cooldown window after load stopped. Check: `cat artifacts/<run-id>/hpa/replica_samples.csv`. Fix: `./scripts/fix.sh HPA-306`. Retry: `./scripts/up.sh --resume hpa_proof`
- [ ] Each card also notes which Attempt (1/2/3) escalation applies if first fix fails

### Task 10.2.3 — Add profile-specific expectations section: what to expect per profile during proof
- [ ] Add **Profile Expectations** section to `docs/troubleshooting.md`
- [ ] `tiny` profile: expect scale-up within 60–120s; cooldown within 3–5 minutes; `maxReplicas=5` reached under moderate load
- [ ] `balanced` profile: scale-up within 45–90s; higher ceiling (7 replicas); suitable for demonstrating graduation from tiny
- [ ] `stretch` profile: scale-up within 30–60s; `maxReplicas=10`; may require extended cooldown window; most realistic of the three
- [ ] Note that actual timing depends on machine load, Docker resource limits, and KinD node capacity
