# Troubleshooting

Failure codes are generated by the HPA proof gate (`gate_hpa_proof`). When a gate fails, `up.sh` prints an actionable hint and the exact next command. `./scripts/fix.sh HPA-30X` applies the canonical fix — up to 3 escalating attempts automatically.

---

## HPA-301 — HPA resource not found

**Symptom:** The HPA object is missing from the namespace after bootstrap.

**Check:**
```bash
kubectl get hpa -n autoscaling-lab
```

**Fix:**
```bash
./scripts/fix.sh HPA-301
```

**Retry:**
```bash
./scripts/up.sh --resume hpa_proof
```

**Escalation ladder:**
| Attempt | Action |
|---------|--------|
| 1 | `kubectl apply -f k8s/hpa.yaml` |
| 2 | Server-side apply + patch `scaleTargetRef.name` to match deployment |
| 3 | Delete + recreate HPA from `k8s/presets/hpa-proof.yaml` |

---

## HPA-302 — Metrics server unavailable

**Symptom:** `kubectl top nodes` fails or returns no data; HPA cannot read CPU metrics.

**Check:**
```bash
kubectl -n kube-system get pods -l k8s-app=metrics-server
kubectl top nodes
```

**Fix:**
```bash
./scripts/fix.sh HPA-302
```

**Retry:**
```bash
./scripts/up.sh --resume hpa_proof
```

**Escalation ladder:**
| Attempt | Action |
|---------|--------|
| 1 | Re-apply `k8s/addons/metrics-server.yaml` + rollout wait |
| 2 | Restart deployment + patch `--kubelet-insecure-tls` arg if missing |
| 3 | Delete + recreate metrics-server from pinned KinD manifest |

> **macOS note:** `kubectl top nodes` may take up to 60 seconds to return its first reading after metrics-server starts. The bootstrap gate retries automatically; if HPA-302 still fires, try attempt 2.

## HPA-303 — CPU requests not set on deployment

**Symptom:** HPA shows `<unknown>/50%` for CPU target; HPA cannot scale without a CPU request on the container.

**Check:**
```bash
kubectl -n autoscaling-lab get deploy sample-app \
  -o jsonpath='{.spec.template.spec.containers[0].resources.requests.cpu}'
```

**Fix:**
```bash
./scripts/fix.sh HPA-303
```

**Retry:**
```bash
./scripts/up.sh --resume hpa_proof
```

**Escalation ladder:**
| Attempt | Action |
|---------|--------|
| 1 | Re-apply `k8s/app/<APP_MODE>/deployment.yaml` + rollout wait |
| 2 | `kubectl patch deploy` to set `cpu: 100m, memory: 128Mi` directly |
| 3 | Apply `k8s/presets/deployment-tiny-safe.yaml` (cpu: 50m guaranteed) |

## HPA-304 — Load generator failed to start

**Symptom:** `start_load` returned non-zero; no load is reaching the app.

**Check:**
```bash
./scripts/load.sh --status
kubectl -n autoscaling-lab get pod load-generator   # for pod mode
```

**Fix:**
```bash
./scripts/fix.sh HPA-304
```

**Retry:**
```bash
./scripts/up.sh --resume hpa_proof
```

**Escalation ladder:**
| Attempt | Action |
|---------|--------|
| 1 | Re-run `load.sh --mode <LOAD_MODE>` |
| 2 | Switch load mode (`pod→host` or `host→pod`); write override to `.state/env-overrides` |
| 3 | Start fallback `kubectl run load-fallback` busybox pod with low rate |

---

## HPA-305 — No scale-up observed during ramp window

**Symptom:** Replica count never exceeded baseline during the `HPA_RAMP_SEC` window.

**Check:**
```bash
cat artifacts/<run-id>/hpa/replica_samples.csv
kubectl -n autoscaling-lab get hpa
```

**Fix:**
```bash
./scripts/fix.sh HPA-305
```

**Retry:**
```bash
./scripts/up.sh --resume hpa_proof
```

**Escalation ladder:**
| Attempt | Action |
|---------|--------|
| 1 | Apply `k8s/presets/hpa-proof.yaml` (CPU target: 20%) + start load |
| 2 | Extend ramp to 300s (`HPA_RAMP_SEC=300`) + increase concurrency to 50 |
| 3 | Lower HPA CPU target to 20% + switch to host load mode |

**Common causes:** CPU target too high for the load intensity; `tiny` profile node already saturated by system pods; Docker resource limits set too low.

---

## HPA-306 — Cooldown not observed within window

**Symptom:** Replicas did not decrease within the `HPA_COOLDOWN_SEC` window after load stopped.

**Check:**
```bash
cat artifacts/<run-id>/hpa/replica_samples.csv
kubectl -n autoscaling-lab describe hpa sample-app-hpa
```

**Fix:**
```bash
./scripts/fix.sh HPA-306
```

**Retry:**
```bash
./scripts/up.sh --resume hpa_proof
```

**Escalation ladder:**
| Attempt | Action |
|---------|--------|
| 1 | Stop load + extend cooldown window to 420s |
| 2 | Patch HPA `scaleDown.stabilizationWindowSeconds=60` + set `HPA_COOLDOWN_SEC=300` |
| 3 | Enforce zero-load + full 420s (7-minute) observation window |

**Note:** HPA-306 does not prevent the scale-up proof from being recorded. The `summary.txt` artifact still notes that scale-up was proved even when cooldown was not observed.

## Profile Expectations

Actual timing varies with machine load, Docker resource limits, and KinD node capacity.

### tiny (maxReplicas: 5)

- Scale-up expected within **60–120 seconds** of load start
- Cooldown expected within **3–5 minutes** after load stops
- Suitable for: laptops, CI, quick demos
- If scale-up takes longer than 120s, check that Docker has ≥2 CPU cores allocated

### balanced (maxReplicas: 7)

- Scale-up expected within **45–90 seconds**
- Higher replica ceiling demonstrates more pronounced autoscaling
- Suitable for: development workstations with ≥4GB available RAM
- Consider `HPA_RAMP_SEC=240` if your machine is slow to warm up

### stretch (maxReplicas: 10)

- Scale-up expected within **30–60 seconds**
- Most realistic profile; most likely to need extended cooldown window
- Suitable for: high-spec machines or cloud VMs with ≥8GB available RAM
- May benefit from `HPA_COOLDOWN_SEC=420` pre-emptively to avoid HPA-306
